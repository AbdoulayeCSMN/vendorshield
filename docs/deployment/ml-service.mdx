Le service ML (`services/ml/`) entraîne et sert les prédictions de risque opérationnel via
**MLflow**. Il est **optionnel** : sans lui, l'application utilise son modèle TypeScript
embarqué.

## Quand l'utiliser

<CardGroup cols={2}>
  <Card title="Sans le service" icon="circle-check">
    Le modèle TypeScript embarqué suffit. Aucune infrastructure à gérer.
  </Card>
  <Card title="Avec le service" icon="server">
    Modèles plus riches, suivi MLflow, validation croisée. Pour les déploiements à volume.
  </Card>
</CardGroup>

## Configuration

```bash
cd services/ml
cp .env.example .env
```

| Variable | Rôle |
|---|---|
| `VS_ML_DATABASE_URL` | DSN lecture seule (`postgresql+psycopg://…`) vers Supabase |
| `VS_ML_API_KEY` | Clé envoyée par l'app dans l'en-tête `X-API-Key` |
| `VS_ML_MLFLOW_EXPERIMENT` | Nom de l'expérience MLflow |
| `VS_ML_MIN_SUPPLIER_HISTORY` | Historique minimal par fournisseur (défaut 8) |
| `VS_ML_PPM_BREACH_THRESHOLD` | Seuil de dépassement PPM (défaut 5000) |

## Déploiement gratuit (Cloud Run)

Le service tourne sur n'importe quelle plateforme Docker. Le meilleur choix **gratuit** est
**Google Cloud Run** : scale-to-zero (gratuit quand inactif), démarrage à froid en
**secondes**, RAM configurable. Le `Dockerfile` **entraîne les modèles au build** — l'image
est prête à servir, et le **serving n'a pas besoin de la base** (l'app envoie les livraisons
dans la requête).

<Steps>
  <Step title="Prérequis">
    Un projet Google Cloud + le CLI `gcloud` installé et authentifié
    (`gcloud auth login`, `gcloud config set project <PROJET>`).
  </Step>
  <Step title="Déployer depuis le dossier du service">
    ```bash
    cd services/ml
    gcloud run deploy Avilyre-ml \
      --source . \
      --region europe-west1 \
      --memory 1Gi \
      --min-instances 0 \
      --allow-unauthenticated \
      --set-env-vars VS_ML_API_KEY=choisis-une-cle
    ```
    Cloud Run construit l'image via le `Dockerfile`, puis déploie. Notez l'URL
    `https://Avilyre-ml-xxxx.a.run.app`.
  </Step>
  <Step title="Brancher l'application (Vercel)">
    ```bash
    ML_SERVICE_URL=https://Avilyre-ml-xxxx.a.run.app
    ML_SERVICE_API_KEY=choisis-une-cle   # identique à VS_ML_API_KEY
    ```
  </Step>
</Steps>

<Note>
**Scale-to-zero** : le service s'éteint quand il n'est pas sollicité (donc gratuit). Le
premier appel après une pause subit un court démarrage à froid ; pendant ce temps l'app
**retombe sur le modèle TS**. Aucune panne.
</Note>

### Entraîner sur vos vraies données

Par défaut les modèles sont entraînés sur des **données synthétiques** au build (suffisant
pour démarrer). Pour entraîner sur votre base, exécutez l'entraînement là où le DSN est
joignable (`VS_ML_DATABASE_URL` défini), puis redéployez l'image — ou planifiez un
ré-entraînement périodique (Cloud Run Job / cron).

## Intégration

L'application appelle le service via `predictViaMlService()`. En cas d'indisponibilité ou
d'erreur, elle **retombe silencieusement** sur le modèle TypeScript — aucune interruption
côté utilisateur.
