VendorShield prédit le **risque opérationnel** par fournisseur — probabilité de **retard
de livraison** et de **défauts (PPM)** — à partir de l'historique des livraisons.

## Architecture

```
Service ML Python (FastAPI + MLflow)  ──HTTP──►  predictViaMlService()
        │                                              │ (repli automatique)
        ▼                                              ▼
   prédiction servie                          Modèle TypeScript embarqué
```

L'application appelle d'abord le service Python ; s'il est indisponible, elle **retombe
automatiquement** sur le modèle TypeScript. L'app fonctionne donc **sans** le service ML.

## Modèle de prédiction

**Probabilité de retard** — régression logistique sur des features calculées en fenêtre
expansive (sans fuite temporelle) :

$$
P(\text{retard}) = \sigma(z), \qquad z = \mathbf{w}^{\top}\mathbf{x}, \qquad \sigma(z) = \frac{1}{1 + e^{-z}}
$$

où $\mathbf{x}$ regroupe le biais, l'OTD antérieur, le retard moyen antérieur, la quantité,
le PPM et la saisonnalité ($\sin$/$\cos$). Les poids $\mathbf{w}$ sont appris par **descente
de gradient** minimisant la log-vraisemblance avec régularisation $L_2$ :

$$
\mathcal{L}(\mathbf{w}) = -\frac{1}{n}\sum_{i=1}^{n}\Big[y_i\log \hat{p}_i + (1-y_i)\log(1-\hat{p}_i)\Big] + \lambda\lVert \mathbf{w}\rVert^{2}
$$

La **contribution** d'un facteur $j$ (driver expliqué à l'utilisateur) est $c_j = w_j \cdot x_j$.

**Dépassement du seuil PPM** — à partir du PPM projeté $\widehat{\text{PPM}}$ (tendance
linéaire) et du seuil $\tau$ (défaut $5000$) :

$$
P(\text{dépassement}) = \sigma\!\left(1{,}7 \cdot \frac{\widehat{\text{PPM}} - \tau}{\sigma_{\text{rés}}}\right)
$$

où $\sigma_{\text{rés}}$ est l'écart-type des résidus de la tendance.

## Cold-start : valeur dès le 1er jour

<Note>
Un nouveau compte n'a pas d'historique. Plutôt que d'échouer, VendorShield utilise un
**modèle global anonymisé** (coefficients agrégés inter-comptes, **aucun identifiant de
tenant exposé**) pour produire une prédiction initiale (confiance plafonnée). Dès que le
compte a assez de données, un modèle dédié prend le relais automatiquement.
</Note>

## Quand les modèles s'entraînent

| Déclencheur | Portée |
|---|---|
| Bouton « Recalculer la prédiction » (fiche fournisseur) | Un fournisseur |
| Bouton « Recalculer les prédictions » (liste fournisseurs) | Tout le compte |
| Cron `POST/GET /api/ml/retrain` | Tous les comptes |
| Nouveau compte sans historique | Modèle global (cold-start) |

Le ré-entraînement batch est **rapide et gratuit** (sans appel LLM, explication
déterministe). Voir [Automatisation](/deployment/automation) pour la planification.

## Confidentialité

Le modèle global n'apprend que des **coefficients agrégés** ; il ne stocke ni n'expose
aucune donnée nominative d'un autre client. Chaque compte ne voit que ses propres
prédictions (RLS).

## Défaillance financière

Une prédiction de **risque de faillite** est disponible par fournisseur (onglet dédié sur
la fiche), alimentée par l'Edge Function `bankruptcy-predictor`.
